## MILESTONE: M11 -- Alerting + Observability Hardening

### Spec Context
All Phase 3 routes already emit structured logs via structlog (M5–M10).  M11 hardens
observability by adding two operator-facing HTTP endpoints that give the diagnostics
shell a real-time snapshot of platform health without the UI computing state locally.

Two new endpoints are scoped for M11:

1. **GET /health/dependencies** — reports the reachability and status of each backend
   dependency: database, queue system, artifact store, and feed health service.
   Each dependency is classified as OK, DEGRADED, or DOWN with an optional detail string.
   The overall_status field summarises: OK if all deps are OK; DEGRADED if any are
   DEGRADED; DOWN if any are DOWN.  This powers the operator diagnostics header badge.

2. **GET /health/diagnostics** — returns a platform-wide operational snapshot:
   queue_contention_count (number of queue classes with active contention),
   feed_health_count (total feed health snapshots in the system),
   parity_critical_count (number of unresolved CRITICAL parity events),
   certification_blocked_count (number of feeds with BLOCKED certification status).
   This powers the diagnostics shell overview panel.

### Key Constraints
- FastAPI app: services/api/main.py — never app.py or any other name.
- Route handlers: services/api/routes/observability.py (new file).
- Pydantic schemas: libs/contracts/observability.py (new file).
- LL-007: Use str="" for DependencyHealthRecord.detail (not Optional[str]).
- LL-008: All handlers use JSONResponse + dict; no response_model=.
- Dependency health checks are always OK in the mock (no real connectivity in sandbox).
- DiagnosticsSnapshot aggregates from mock repos injected via DI.
- No external network calls in route handlers or mocks.

### New Contracts (libs/contracts/observability.py)
